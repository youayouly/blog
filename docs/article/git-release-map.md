# Git 发布流水线：从本地改动到 Vercel Release

这篇不是 Git 命令大全，而是这个站点自己的发布地图。以后文章、封面、批量发布、主题样式和 Vercel 构建混在一起时，可以先按阶段回看：这段时间到底是在写内容、补资源、修发布工具，还是在处理线上构建。

## 1. 工作区：先看清楚改了什么

工作区是本地文件真实发生变化的地方。写文章、生成封面、改组件、调样式，最先都会落在这里。

常用检查：

```bash
git status --short
git diff
```

重点不是急着提交，而是先分清楚两类改动：一类是这次任务必须带走的内容，另一类是日志、调试脚本、备份文件、临时测试文章。后者如果被带进仓库，后面很容易让 VuePress 或 Vercel 读到不该读的文件。

## 2. 暂存区：把同一件事打包

暂存区可以理解成提交前的篮子。比如一篇文章引用了三张新封面，那么文章文件、列表页和三张图片要一起进入暂存区。只提交文章、不提交图片，本地可能还能看到，云端就会直接缺资源。

常用检查：

```bash
git diff --cached --name-only
git ls-files docs/.vuepress/public/gallery/
```

这次站点踩过的坑就是：列表已经引用了新的封面图片，但图片文件没有随提交一起进入仓库。Vercel 在云端只看 Git 里的文件，所以“本地能看见”不等于“线上能构建”。

## 3. Commit：让每段工作能被回看

Commit 是时间线里的一个节点。好的提交信息不一定长，但要说明这次发布解决了哪类问题。

比较容易回看的写法：

```text
新增 openclaw、ai infra (批量发布 2 篇文章)
修复生成封面随文章一起发布
增强文章发布封面兜底
更新文章推荐与发布流水线说明
```

这样之后看 `git log`，就能判断某个时间段是在发内容、修发布流程，还是补构建兜底。

## 5. 同步远端：避免本地和 GitHub 分叉

如果 GitHub 上已经有新提交，而本地还没同步，直接推送会被拒绝。这不是坏事，它是在提醒我们：先把远端变化合并进来。

常见流程：

```bash
git pull --rebase origin main
git push origin main
```

如果本地还有未提交改动，先不要硬推。应该把当前任务提交好，或者临时 `stash`，再 rebase。这里最重要的是不要覆盖别人的提交，也不要把还没验证的本地状态强行推上去。

## 6. Push 与 Vercel Release：云端只认仓库状态

Push 到 GitHub 后，Vercel 会按仓库里的代码重新构建。它不会读取电脑里未提交的图片、临时文章或 `.env` 文件。

Vercel 失败时，先看三件事：

1. 报错路径指向的文件是否真的被 Git 跟踪。
2. Markdown 或组件里引用的本地资源是否存在于 `docs/.vuepress/public/` 下。
3. 临时备份文件是否被 VuePress 当成页面收进路由。

构建前跑一遍：

```bash
npm run build
```

本地 build 过了，再 push，心跳会稳很多。

## 7. 批量发布：文章、封面、列表是一个事务

批量操作最容易出现“文章提交了，图片没提交”或“列表改了，正文被删了”的半成品状态。理想流程是把它当成一个事务：

1. 生成所有封面。
2. 检查每篇文章都不再使用占位图。
3. 一次性更新文章正文、文章列表和封面文件。
4. 检查 `git status`，没有真实改动就不提交。
5. 有真实改动时再提交。
6. 本地构建验证。
7. 推送触发 Vercel。

如果其中一步失败，就不要把半成品推上去。宁可保留在待处理列表，也不要让线上构建替我们发现缺图。

## 8. 用时间线回看每段时间在做什么

以后可以把 Git 历史粗略分成几类：

- `Article`：新增、更新、删除文章。
- `Assets`：封面图、相册、静态资源。
- `Build`：VuePress、Vite、Vercel 构建修复。
- `Workflow`：发布按钮、批量操作、自动生成封面。
- `Style`：主题、浅色 / 深色模式、页面视觉。

About 页里的动态时间线就按这些类别记录。这样提交多了也不会乱：先看时间线知道阶段，再进 Git log 找具体提交。
